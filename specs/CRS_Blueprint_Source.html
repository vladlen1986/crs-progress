Multi-Property Tenancy
Locked
Decision
Company = Casino for MVP. Schema designed v2-ready for multi-property.
Architecture
Every business Data Type carries both company and property fields from day one. A Property Data Type exists; one Property is auto-created per Company on signup. Multi-property UI hidden behind feature flag — user sees a single "Casino Settings" page. v2 = flip the flag, add property switcher UI, zero migration needed.
2
Role & Permission Granularity
Locked
Decision
Custom Role Data Type (per Company) + fixed Permission Option Set (global) + optional extra_permissions on User for edge cases.
Architecture
Permission = Option Set with global, fixed codes (report.create, report.send, task.cancel, etc.). Cannot be renamed or added at runtime — synced with privacy rule logic. Role = Data Type, tenant-scoped. Admin creates any role name ("Pit Boss", "Shift Lead"), picks permissions from the Option Set. User has one role + optional extra_permissions list. Privacy rule: Current User's role's permissions contains "report.send" OR Current User's extra_permissions contains "report.send". Role edits propagate live to all users with that role.
3
GDPR / Data Retention
Locked
Decision
Hybrid — Anonymize PII holders, never delete audit records, soft-delete config lookups.
Rules by Data Type
PII holders (Employee, Guest, User) → is_anonymized field; scrub first_name/last_name/email/phone/doc_id/avatar on erasure request, preserve foreign key links.
Audit records (Report, Task, RFI, ReportResponse, ReportComment, ReportRevision, ActivityLog) → never deleted, never anonymized.
Config / lookup (Tag, Department, Section, Location, GameType, ReportType, Role) → soft-delete with is_active=false.
Attachments with PII (photos, ID scans) → full delete allowed, admin-confirmed.
Draft reports → full delete allowed before Submit by the creator.

Subscription Tiers + Attachments
Locked
Decision
5 pre-built subscription tiers, each bundling a set of modules. Videos stored as external URL references (never uploaded to Bubble).
Tiers
Starter → Standard → Professional → Executive → Enterprise. Each Tier has included_modules list. Core modules (Users, Roles, Notifications, Casino Settings) always active regardless of tier. Tiers are a Data Type you seed; Company has a single tier field pointing to one Tier.

Security Rules
Non-Negotiable — Multi-Tenant Isolation
  • Every Typesense index MUST have company_id as a required, filterable field.
  • Omnisearch preserves Bubble privacy rules automatically — this is why we chose it over Zeroqode's Algolia V2.0 plugin.
  • Prerequisite: Data Type privacy rules must be correctly configured (Current User's Company = Thing's Company) BEFORE enabling search. If privacy rules are wrong, search inherits wrong rules.
  • Security Review checklist: verify privacy rules on Report, Task, RFI, Employee, Guest, Comment before enabling search sync.
Sync Strategy
  • Create / Update → backend workflow pushes record to Typesense via Omnisearch action.
  • Anonymize Employee/Guest → backend workflow re-indexes with scrubbed fields.
  • Soft-delete config lookup → no index change (no index for these anyway).
  • Hard delete (attachments, draft reports) → backend workflow removes from index.
  • Nightly reconciliation workflow — compare Bubble records against Typesense, fix drift. Critical for long-term correctness.

Data Model — Principles
Draft
Rules that every Data Type follows across the CRS app.
Core Principles
  • Every tenant-scoped Thing has company AND property fields — company for multi-tenant isolation, property for v2-ready multi-property support (Decision #1).
  • Every Thing has audit fields — created_date, modified_date, created_by, modified_by.
  • Privacy rules enforce isolation — Current User's company = Thing's company, Current User's property = Thing's property.
  • Permission checks use custom Role + fixed Permission option set (Decision #2) — Current User's role's permissions contains X OR Current User's extra_permissions contains X.
  • PII holders (Employee, Guest, User, ApplicantAccount, Applicant) have anonymization fields (Decision #3) — is_anonymized, anonymized_date, anonymized_by, anonymization_reason.
  • Config / lookup Data Types have is_active for soft-delete — Tag, Department, Section, Location, GameType, ReportType, Role.
  • Use Junction Data Types for many-to-many relations — no heavy List fields on high-traffic Things.
  • Split heavy Things into Header + Detail — list views query header only, details lazy-loaded.
  • Counters are atomic — dedicated Counter Data Type, never compute via :count + 1.
  • Module gating on Company's tier (Decision #4) — every nav item / page / workflow checks Company's tier's included_modules contains this module unless module is_core.
  • Public-access data types are explicitly flagged v20 — only whitelisted fields on JobOpening, JobOpeningTranslation, Company, Property, Division/Section/Department/Position are visible to anonymous (not-logged-in) users. See "Public-Access Privacy Pattern" below for full matrix.
  • ApplicantAccount is a separate identity space v20 — does NOT use Bubble's User type. Has custom auth via API Workflows + session tokens. Explicit exception to "every DT has company + property" (ApplicantAccount is globally scoped identity).
  • Plain text only — no rich text editor anywhere in the app v20. All long-text fields (job descriptions, post bodies, comments, policy content, notes) store plain text with \n line-break preservation. Bubble renders via white-space: pre-wrap on display elements. No HTML, no Markdown, no WYSIWYG. Reduces XSS attack surface, eliminates sanitization complexity, keeps copy-paste across modules lossless, and enforces visual consistency (Info Board post body reads identically to Employee note reads identically to Job Opening description).
Field Convention Shorthand
To keep this doc readable, individual Data Type specs below show only the business-specific fields. Every Data Type in the app has id, company, property, created_date, modified_date, created_by, modified_by unless noted otherwise. PII holders additionally have the 4 anonymization fields. Config lookups additionally have is_active. When a spec table shows these fields explicitly, it's for emphasis — they're always present on tenant-scoped Things.

Security review checklist — BEFORE adding any field to public DTs
  • 1. Default to hidden. In Bubble, Privacy Rules default to "all fields hidden" for Everyone. New fields appear hidden — they must be explicitly checked in the public rule to be visible.
  • 2. Review on every schema change. Any PR that adds a field to JobOpening, Company, Property, Division, Section, Department, Position, or JobOpeningTranslation must include a documented review of whether the new field is safe for public exposure.
  • 3. Never expose internal FKs. hiring_manager, recruiter, internal_notes, headcount_target, headcount_filled, created_by, modified_by, screening logic, audit fields — all MUST stay blocked.
  • 4. Denormalized flags only. Public rules should never use filtered searches of related records (expensive + leakable). Use denormalized boolean flags on parent records.
  • 5. Test as anonymous. Every public page must be tested in an incognito window before release.

Threat model — known attack vectors
  • Credential stuffing: attacker uses leaked password lists against public.applicant.login. Mitigation: rate limit + CAPTCHA after 3 failures + generic error messages ("Invalid credentials" — don't reveal whether email exists).
  • Application spam: bot submits thousands of fake applications. Mitigation: CAPTCHA + honeypot + rate limit + required file attachments (CV) blocks most bots.
  • Draft abuse: attacker fills draft storage with junk. Mitigation: max 1 active draft per IP per 24h + 30-day auto-cleanup of unsubmitted drafts.
  • Tenant enumeration: attacker probes for valid company slugs. Mitigation: generic 404 for invalid slugs, rate limit on public.opening.list_by_company.
  • Session hijacking: attacker steals token from URL/cookie. Mitigation: token expiry (24h default / 30d "remember me"), rotate on every login, invalidate on password change, HTTPS only.
  • PII scraping: attacker tries to enumerate Applicants. Mitigation: Applicant records NOT exposed via any public WF. Portal access only via session_token, scoped strictly to account_id.

Dedupe & Batching
Dedupe Rule — One notification per (user + event + entity)
If a user appears on the recipient list multiple times for the same event on the same entity (default recipient AND subscriber, for example), the list is deduplicated before creating Notifications. The user sees exactly one.
Batching Logic
  • Grouping key: (recipient + event_type + module + time window).
  • Window: event type's Batching Window (min) attribute.
  • Batch hit: append entity_id to related_entity_ids, increment batch_count, update modified_date. Do NOT re-schedule email.
  • Batch miss: create new Notification. Schedule email for now + batching_window.
  • Critical bypass: Batching Window = 0 → never batched, immediate send.
Why Batching Matters at Scale
WU & Email Deliverability Impact
A bulk import of 50 reports triggers report.created 50 times. Without batching: 50 Notification records per recipient + 50 emails. With batching (15-min window): 1 Notification + 1 email saying "12 reports were created in Cage department in the last 15 minutes." WU cost drops 98%. SendGrid won't flag the send as spam. Raw 50-email bursts can trigger deliverability throttling across the whole tenant.
Delivery Workflow
Three backend workflows form the pipeline. Check-on-fire guard ensures idempotency and no duplicate sends.
1. trigger_notification (entry point)
Called by every module when an event fires.
Parameters: event_code, entity_type, entity_id, actor, optional extra_context.
  • Load OS - Notification Event Type by code. If Is Active = false → exit silently.
  • Resolve default recipients via event-specific resolver.
  • Search Subscription for this entity → add non-muted subscribers.
  • Exclude actor. Dedupe list.
  • For each recipient → schedule deliver_to_user (keeps calling workflow fast, avoids timeout on large recipient lists).
2. deliver_to_user (scheduled, per-recipient)
  • Load recipient's NotificationPreferences for this event.
  • Determine active channels (respect Allow Opt-Out, skip muted non-critical).
  • If no channels active → exit (user fully muted, never create record).
  • Batching check — search for existing unread Notification with matching grouping key within window.
  • Batch exists: update existing (append entity_id, increment count, refresh modified_date). Do not re-schedule email.
  • No batch: create new Notification. If email in channels → set email_scheduled_send_date + schedule send_notification_email.
3. send_notification_email (scheduled)
  • Load Notification. If null/deleted or state != Unread → exit.
  • Check-on-fire guard: if email_scheduled_send_date != this scheduled time → skip (defensive; shouldn't happen with this architecture).
  • Load EmailTemplate by event_type's template_key.
  • Substitute placeholders from Notification + related entities.
  • If batch_count > 1 → render "N events" version. Otherwise single version.
  • Send via SendGrid plugin action.
  • Update Notification: email_sent_date, email_status = Sent (or Failed on error).
Why This Is Robust
  • No reschedule complexity — each email scheduled once, checked on fire. Avoids Bubble scheduler race conditions.
  • Idempotent — re-firing a trigger doesn't create duplicates (batch check catches it).
  • Fast entry point — delegates per-user work to scheduled child workflows.
  • WU-efficient — writes only for users who actually want the notification.
  • Graceful degradation — deactivated event types exit cleanly.

Design rules
  • 2nd user referenced (assigned, mentioned, sent, shared, role_assigned) → use related_user + related_user_snapshot.
  • 2nd record referenced (comment, attachment, response) → use the related_subject_* trio.
  • Value change tracking (edited, permission_granted) → use field_changed + from_value + to_value.
  • Bulk/quantified operations (export, bulk edit, import) → use affected_count.
  • Multi-recipient events (sent to 5 people) → write 5 separate rows, one per recipient. Simpler queries, no list fields.

Subject lookup gotcha
subject_id is a text pointing to another record. When the UI does Do a search for Report where unique_id = ActivityLog's subject_id, Bubble applies the Report's privacy rules — not ActivityLog's. If the user can't read that Report, the lookup returns empty. The subject_snapshot still renders (since it lives on the ActivityLog row). Click-through returns empty. This is expected cascading behavior — safe, no leak.

Gaming Date v21
Atomic operational period — the real "business day" for a property. Opens + closes explicitly by power user, typically spans a 24-hour shift cycle. Gaming Date is the finer-grained classification field below FiscalWeek; every event record carries both.
Why Gaming Date exists
Casinos don't operate on calendar days. A shift that opens at 18:00 and closes at 06:00 crosses midnight but is a single operational period. "Gaming Date 2026-04-21" means "the gaming cycle that opened on 21 April" — regardless of which calendar date its close ends up on. All reports, tasks, malfunctions, and shift-log entries during that cycle tag to this single Gaming Date, which simplifies daily-brief and shift-log rollups.

No auto-open (deliberate)
Unlike FiscalWeek (which cascades open automatically when the previous closes), GamingDate must be opened explicitly each cycle. The gap between one GamingDate's close and the next's open represents the handoff window. During this gap, the property has no current GamingDate. Records created during the gap attach to the most recently closed GamingDate as a fallback — better than null. A "Gaming Date Not Open" warning banner appears in the app until the next one is opened.
Close cascade — what happens
  • Step 1 — Finalize close: closed_at = now, closed_by = actor, status = closed. Write ActivityLog.
  • Step 2 — Classification sweep: find records where gaming_date is null AND created_date falls between this GamingDate's opened_at and closed_at. Set their gaming_date FK.
  • Step 3 — Check if this closes the parent FiscalWeek: if this was the last GamingDate of the week (i.e. now ≥ parent FiscalWeek's planned_end), the FiscalWeek's grace period starts or its auto-close condition may fire.

Fiscal Week v21
Weekly reporting period. Pre-generated for 5 years at Property setup based on admin-configured close day + time. Weeks auto-cascade — closing Week N automatically opens Week N+1. Year-boundary rule creates short first/last weeks per year so every fiscal year aligns to Jan 1.
Architecture — cached grouping field, not source of truth
Every event Data Type (Report, Task, RFI, Malfunction, ShiftLog, InfoBoard post, etc.) carries a fiscal_week FK populated at create-time. This is a cached classification, not the primary time reference — each record still has its real created_date and event-specific dates. If the FiscalWeek DT is misconfigured or deleted, records stay intact via their real timestamps; they're just "unclassified" until retroactive classification runs.

Pre-generation algorithm
  • Trigger: runs when admin first configures Property's week_close_day + week_close_time. Generates 5 years ahead of FiscalYear + FiscalWeek records.
  • Rolling extension: scheduled backend workflow runs annually on 1 October at 02:00 property time. If the latest-generated year is less than 5 years ahead, generate the next year.
  • Week 1 of each year starts at Jan 1 at week_close_time. May be shorter than 7 days — ends at the next week_close_day occurrence at week_close_time.
  • Middle weeks are exactly 7 days, starting from previous week's planned_end.
  • Last week of year is cut short: planned_end = Jan 1 of next year at week_close_time. May be 1–6 days long depending on where Dec 31 falls relative to week_close_day.
  • Year assignment: a FiscalWeek belongs to the fiscal_year of its planned_start. Cross-year weeks (short Dec week) belong to the year that begins them, not the year they end in.
  • Property go-live: if the property's go-live date is mid-year, the first FiscalWeek's planned_start = go-live timestamp (not Jan 1). Earlier weeks of that year simply don't exist for this property.

Close cascade — what happens when a week closes
  • Step 1 — Finalize close: set closed_at = now, closed_by = actor, status = closed. Write ActivityLog entry.
  • Step 2 — Classification sweep: find all records (Report, Task, RFI, etc.) where fiscal_week is null AND the record's primary date falls in [planned_start, planned_end]. Set their fiscal_week FK to this week. Typical case: zero records found (classify-on-create handles most).
  • Step 3 — Cascade open next week: find the next FiscalWeek for this property (by planned_start). Set its status = current, opened_at = now, opened_by = same actor.
  • Step 4 — Trigger WeeklyReport generation: schedule backend workflow generate_weekly_report with this FiscalWeek's ID. Runs async; result linked back via FiscalWeek.weekly_report (added later when WeeklyReport DT lands).
  • Step 5 — Notify: send in-app + email notifications to fiscal_week.close_digest permission holders.
Auto-close safety net
If a power user forgets to close the week, the system auto-closes after admin-configured grace period. Configuration lives in Property settings:
  • week_close_time — nominal time (e.g. 11:00)
  • week_close_grace_hours — default 6, admin-tunable 1–72
Notifications to users with fiscal_week.close permission:
  • At planned_end: "Week N is due to close now. Close manually or it will auto-close in {grace} hours."
  • 1 hour before auto-close: "Week N will auto-close in 1 hour. Close manually to record your name."
  • At auto-close: "Week N was auto-closed because no manual close occurred." Sent to close-permission holders + admins.
Classification rule — the "latest open week" shortcut
  • On create: every event record gets its fiscal_week FK set to the latest current (or reopened) FiscalWeek for the record's property. Simple, no date-math at write time.
  • Grace-period records attach to the still-open week. Reports filed between planned_end and the actual close fall into Week N (since Week N is still the latest open). This matches operational reality: weeks don't "end" until humans close them.
  • Backdated entries: if a user files a report with event_date from an already-closed week, the fiscal_week still defaults to the latest open. Admin can explicitly re-assign via an audit-logged workflow if needed (future: fiscal_week.reassign permission).
  • No open week? Should not happen in normal operation (cascade auto-opens). If it does (property offline, all weeks closed, pre-gen failed), records get fiscal_week = null. Retroactive classification sweep runs on next close.
Historical migration (existing 4-year data)
  • Step 1: Pre-gen FiscalYear + FiscalWeek records for the 4-year history. All weeks status = closed, closed_by = system-user, auto_closed = no, closed_at = planned_end (approximated).
  • Step 2: CSV import records. For each: compute fiscal_week by matching the record's primary date to the FiscalWeek whose [planned_start, planned_end] contains it.
  • Step 3: Generate current + next 5 years of FiscalWeeks. Set the current one to current status. Operations begin.

Presence System v17.1
Real-time availability indicator separate from account status. Shows coworkers when a user is available, away from keyboard, in focus mode, or signed out.
Distinction from Account Status
Account Status (Active / Inactive / Anonymized) — set by admin action; controls whether user can log in at all.
Presence (Online / Away / Busy / Offline) — computed from activity + manual override; signals current availability for collaboration (@mentions, task assignment hints, sidebar avatars). An Active user can be Offline (signed out). An Inactive user is always shown as Offline.
Storage
  • Two fields on User: presence_status (OS - Presence Status) + presence_override_until (date, nullable).
  • Stored on User (not Session) so privacy rules stay simple — same-company users can read each other's presence without touching Session (which has blocked session_token).
  • presence_override_until only visible to self + admins — prevents leaking intent ("Marco set himself busy until 17:00").
Computation Logic
If presence_override_until is set AND > now → use manual presence_status
Else (auto-compute):
  • Has Active Session with last_activity < 5 min ago → Online
  • Has Active Session with last_activity 5–30 min ago → Away
  • Otherwise → Offline
Manual Override UX
  • User clicks presence dot on their avatar in sidebar → popover with 4 options.
  • Online → clears override, reverts to auto.
  • Away / Busy → sets presence_status + presence_override_until = now + 8h (or "end of shift" per Scheduling module).
  • Appear Offline → sets override; user appears offline to coworkers while remaining logged in.

Presence writes can explode into thousands per user per day if naively implemented. Two throttles required:

  • Method: TOTP (Time-based One-Time Password). Compatible with Google Authenticator, Authy, 1Password, Microsoft Authenticator.
  • Plugin: Zeroqode "Two Factor Authentication" or equivalent TOTP plugin. One-time cost ~$10–15.

  • Disable flow: User clicks Disable → enters current password → 2FA removed. Writes ActivityLog mfa_disabled.

  • Admin rescue: Admin with admin.edit_user can force-disable 2FA for a user who lost both device + backup codes (out-of-band verification responsibility).

Overview
Employee is the canonical person record. Every business event (reports, tasks, incidents, RFI, surveillance entries, RFID taps) ultimately attributes to an Employee. User = login identity only; Employee = the human. PII lives here, strictly gated.
Core architectural decisions
3-level org hierarchy: Division → Section → Department, all mandatory, seeded on Company signup.
Hybrid multi-property: primary property/dept/position/manager live directly on Employee (95% case). Optional EmployeeAssignment records exist only when employee covers additional properties.
Two identifiers: crs_employee_id (system-generated, immutable, ACM-000247) + company_employee_number (HR manual, for legacy HR/payroll systems).
No hard delete: same pattern as User Management. Employees are anonymized, never deleted. Anonymization requires separation first.
Rehire preserves history: same Employee record reactivated. All warnings, leaves, reports, notes, and history stay linked. EmployeeHistory tracks tenure boundaries.
Sensitive PII strictly gated: national ID, passport, gaming license numbers visible to HR + Admin only via privacy rules.

Leave Status & Separation Topic 3
Leave is temporary (employee still employed). Separation is permanent (employee gone). Re-hire uses the same Employee record to preserve full history.
Leave workflow
  • HR-only assignment in MVP. Employee self-request deferred to Leave Management module.
  • Setting leave: sets leave_status + leave_status_until. employment_status stays Active.
  • Nightly workflow finds leave_status_until ≤ now and clears the leave.
  • Notifications: manager + section head + HR on start and end.
  • Edge cases: tap during leave is logged but flagged for HR review; report/task assignment warns "Employee is on leave" but doesn't hard-block.
Separation workflow
  • Typed SEPARATE confirmation required. Reason (OS) + date + notes + rehirable (yes/no).
  • Can be initiated by: Admin (always), HR Manager (permission employee.separate), Department Head (permission employee.separate_own_department).
  • On separation: User.is_active → false; sessions revoked; RFID card deactivated; employment_status = Separated; leave_status cleared.
  • Active tasks/reports stay linked. HR cleanup dashboard lists them for reassignment — does not block separation.
  • Reversible within 30 days by Admin only (data-entry correction path).
Separated → Archived (nightly)
  • 30 days after separation: employment_status → Archived.
  • Hidden from default employee directory. Visible via "Show archived" toggle.
Re-hire — same record, preserves full history
  • Admin clicks "Rehire" on an Archived or Separated employee (only if rehirable = yes).
  • Required: new hire_date, new department, new position, new manager.
  • employment_status → Active; separation fields cleared; hire_date = new tenure start; original_hire_date unchanged.
  • All linked records retained: past reports, warnings, notes, documents, certifications, history.
  • New RFID card issued (old card stays Deactivated in DB). User account reactivated with forced password reset + 2FA re-enrollment.
  • EmployeeHistory records a new tenure row.

2. Privacy rule uses join: Thing is accessible iff Search Applicants where account = Thing AND company = Current User's company :count > 0 AND Current User has applicant_account.view. 3. Admin tools (anonymize account, reset password): only invokable from within an Applicant's profile page (which already has company context). No global "user admin" screen.
4. Audit: every applicant_account.* access must be logged with actor_company so cross-tenant access attempts are traceable.

🔐 Session token security checklist v20 audit-refinement
Because ApplicantAccount uses custom auth (not Bubble's User type), every security property Bubble handles automatically for internal Users must be manually enforced. Required:

Token generation: cryptographically random UUID v4 (Bubble's Generate random string with "UUID" type). Minimum 128 bits of entropy. Never use incrementing IDs, timestamps, or predictable values. Store as text, index for fast lookup.
Token storage in browser: store in httpOnly cookie where possible (Bubble: set via custom headers on login workflow's response). If Bubble version doesn't support httpOnly response headers, fall back to a short-lived localStorage value. Never store in URL query params (leaks via referer + server logs + browser history). Never embed in page HTML.
Cookie flags (when cookie mode used): Secure (HTTPS only — always on in production), SameSite=Lax (blocks cross-site request forgery on state-changing endpoints while allowing normal portal navigation), HttpOnly (blocks JS access — prevents XSS-based token theft). Set via Bubble's "Set cookie" action with explicit flags, or via the API response headers.
CSRF protection: all portal API workflows MUST verify an anti-CSRF token in addition to session_token. Pattern: on login, generate both session_token (cookie) + csrf_token (client-readable localStorage). Every state-changing portal workflow requires BOTH in the request; server checks they match the same ApplicantAccount. GET-only reads (dashboard, application detail) bypass CSRF check. Public workflows (careers page, submit) use CAPTCHA instead of CSRF tokens.
Token expiry: session_token_expires = login + 7 days (MVP). Sliding window — every authenticated request extends expiry by 7 days if within last 24h of validity. Hard cap: 30 days from initial login — after that, force re-login even if active.
Rotation triggers: rotate session_token on (a) successful login, (b) password change, (c) email verification. On password reset via email link: invalidate all existing tokens for that account (forces logout everywhere).
Revocation: nightly scheduled workflow clears session_token + session_token_expires on rows where expiry has passed. Admin "Revoke all sessions" action in account admin UI clears token immediately.
Rate limits on auth endpoints: login/signup/reset: 10/hour per IP, 5/hour per email. Lockout after 10 failed logins in 1h — 30-minute timer, CAPTCHA required to unlock.
Audit: log every login (success + failure) with IP + user-agent to ActivityLog. Failed-login-streak > 3 from same IP → notify security admin.
What NOT to do: never transmit session_token in URL params, never log session_token to ActivityLog (store hash of it for correlation if needed), never email the token, never expose in API response debug messages.

Transition rules
  • Forward movement: drag-drop in Kanban or "Move to Stage" dropdown in Directory. Recruiter/Hiring Manager only (permission-gated).
  • Backward movement: allowed for Admin + Hiring Manager. Reason text required. Logged as override in ActivityLog.
  • Terminal states: Hired and Rejected/Withdrawn can still be reversed by Admin within 30 days (mirrors Employee separation-reversal window). After 30 days, requires a new application.

  • Rejection → Saved for Later: allowed if applicant opted into talent pool. One-click transition.
  • Every transition: creates ApplicantStageHistory entry + ActivityLog entry + relevant Notifications + updates Applicant.stage + Applicant.stage_entered_date + recomputes retention_category and scheduled_deletion_date (see Retention recompute matrix below).

Retention recompute — edge cases
  • Reversal within 30 days (Admin undoes Hire/Reject): retention_category reverts + scheduled_deletion_date recomputes from new stage. If a record was mid-anonymization grace, reversal restores full retention horizon.
  • GDPRSettings edit (HR changes retention_rejected_days from 90 → 180): backend workflow on GDPRSettings save triggers bulk recompute of scheduled_deletion_date for all affected Applicants in that company. WU cost: one "make changes to list" per retention_category.
  • Past-due at time of recompute: if new scheduled_deletion_date is already in the past (e.g. retention shortened), nightly anonymization picks it up at next run — no immediate action in stage workflow. Prevents surprise mid-day anonymizations triggered by a config change.

2. Create EmployeeAssignment (primary property assignment).
3. Copy all ApplicantDocuments to EmployeeDocuments.
4. Update Applicant in one action: set stage = Hired, linked_employee = Employee, stage_entered_date = now. Bidirectional link now complete.
5. Increment JobOpening.headcount_filled (use "Make changes to a list" with +1 atomic expression — not a read-then-write, to avoid race with other concurrent hires on same opening).

Transactional integrity + reconciliation
Bubble doesn't have true DB transactions. If a later step fails, earlier records already exist. Mitigations:

1. Atomic bidirectional link at creation: step 1 sets Employee.hired_from_applicant in the same create action. Prevents the orphan case where Employee exists without the back-link.
2. "Only when" guards: every step from 2 onwards has Only when step 1's result is not empty and chains from the previous step's success. Failures short-circuit remaining steps, log to a ConversionFailure audit row (timestamp + applicant + failed_step + error_text).
3. Nightly reconciliation — both directions:
    • Forward: Employees where hired_from_applicant is not empty AND hired_from_applicant's stage ≠ Hired → flag for HR (forward-link set but applicant not Hired).
    • Backward: Applicants where stage = Hired AND linked_employee is empty → flag for HR (Hired but no employee).
    • Mismatch: Applicants where linked_employee is not empty AND linked_employee's hired_from_applicant ≠ this Applicant → flag for HR (crossed wires from two concurrent conversions — rare but possible).
4. Dashboard widget: "Conversion integrity" card on HR dashboard shows count of flagged records. Daily email if count > 0.
5. Repair action: HR can manually relink from the flagged-record screen — sets missing FK, logs ActivityLog applicant.conversion_repaired.
Duplicate Detection & Merge Topic 9
Same person often applies to multiple openings, or submits twice by accident. Auto-detection at submit + HR-initiated merge workflow. Never hard-deletes (CRS GDPR policy: anonymize, don't delete).
Auto-detection at submit
On every application submit, backend workflow runs:
Search Applicants where (email = submit.email OR phone = submit.phone) AND company = submit.company AND submitted_date > (now − 180 days) AND id ≠ submit.id.
If results > 0: flag new Applicant with possible_duplicates (list of IDs). Recruiter dashboard shows "⚠ Possible duplicate of {name}" badge.
Same person different openings: NOT treated as duplicate — each opening's application is distinct. Auto-flag is advisory only.
Same person same opening: blocked at submit — "You've already applied to this role. Check your application status."
Merge workflow (HR-initiated)
Requires applicant.merge_duplicates. HR picks winner (keep) + loser (archive).
On merge:
1. Set loser.is_duplicate_of = winner.id.
2. Copy loser's ApplicantNotes → winner (with note: "Merged from duplicate {loser.reference_code}").
3. Copy loser's ApplicantRatings → winner (preserve rater + dimension; if duplicate dimension, keep the more recent).
4. Copy loser's ApplicantMessages → winner (preserve direction + dates).
5. Copy loser's ApplicantDocuments → winner (physical file duplicates).
6. Copy loser's ApplicantStageHistory → winner (preserve timestamps; shown as separate timeline).
7. Set loser.stage = Withdrawn + reason = "Merged into {winner.reference_code}".
8. Set loser.is_anonymized = yes immediately (PII scrubbed — but record preserved for audit).
9. Log ActivityLog: applicant.merged with winner + loser IDs in related fields.
10. Notify winner's ApplicantAccount (if any) — "Your duplicate applications have been consolidated."
Why anonymize loser instead of hard-delete?
Per Decision #3 (CRS GDPR policy): audit records never deleted. Merged duplicate = audit record. The merge action itself must be reconstructible (ActivityLog links point to actual records). Hard-delete would orphan those links. Anonymization scrubs PII while preserving the skeleton for audit integrity. This also aligns with retention rules — loser's data still respects the same retention periods as any Applicant.
🔁 Merge idempotency guard v20 audit-refinement
HR double-clicks "Merge" or two HR users act simultaneously on the same merge suggestion. Without guards, child records get copied twice. Protections:

Step 0 (pre-flight, runs first): abort if loser.is_duplicate_of is not empty (already merged). Return "This applicant is already merged into {winner.reference_code}". No further steps run.
Step 0.5: abort if winner.is_duplicate_of is not empty (winner was itself merged elsewhere — refuse, HR must pick the current primary).
Transactional marker: step 1 sets loser.is_duplicate_of = winner.id FIRST, before any child copies. This flips the pre-flight check's answer — a second invocation aborts at step 0. The marker is the lock.
Child copy idempotence: each child copy step checks if the child already exists on winner with source-marker note (e.g. ApplicantNote copies always include "Merged from {loser.reference_code}" in the body — step 2 searches for that phrase first and skips copy if found). Belt-and-suspenders.
Unique constraint on merge ActivityLog: event body includes loser.id + winner.id — unique constraint on (event_code, loser_id) prevents double-log from parallel executions.
UI: Merge button sets is_merge_in_progress custom state = yes on click; button disabled + spinner shown until workflow completes or errors. Prevents double-click on slow connections.

1. Candidates: OnboardingTemplate WHERE company = X AND is_active = yes AND (property is empty OR property = opening.property) AND (matches_employment_type is empty OR matches_employment_type = employee.employment_type).

Checklist Auto-Create Workflow Topic 5
The backend workflow that fires at the end of the Conversion Wizard (after Employee is created). Resolves the matching template, snapshots its tasks, resolves assignees per task, creates all records. Idempotent — safe to retry.
Trigger conditions
Fires after Conversion Wizard Step 3 finalize — step 8 of the 11-step Employee-creation backend workflow. Parameters passed: employee, applicant, opening. Also manually invokable by HR ("Create Onboarding Checklist" button on Employee profile).
Workflow steps
1. Idempotence check: skip if OnboardingChecklist where employee = X AND status = in_progress exists.
2. Template match (see algorithm in Templates section above). If no match: log, notify HR, stop — no checklist created.
3. Create OnboardingChecklist: name from template, start_date = employee.hire_date, status = in_progress, applicant + employee FKs set, tasks_total counted from template's active task templates.
4. For each OnboardingTaskTemplate in template (where is_active = yes), in sort_order:
  a. Resolve assignee per assignee_role:
    • hr_owner → first User with role "HR Manager" in company (fallback: Admin).
    • dept_head → employee.department.head_user (fallback: HR Manager).
    • direct_manager → employee.manager.user (fallback: HR Manager).
    • it_admin → first User with role "IT Admin" (configured on Company).
    • employee_self → employee.user (may be null if User not yet created — task stays assignee-null until User creation triggers re-resolution).
    • specific_user → template.assignee_specific_user (error if inactive — reassign to HR Manager).
  b. Compute due_date = checklist.start_date + template.due_offset_days.
  c. Create OnboardingTask with snapshot of title/description/category/is_blocking/requires_attachment from template.
  d. Send notification onboarding.task_assigned to assignee (unless self).
5. Compute target_completion_date = start_date + max(task due_offset_days) + 3-day grace.
6. Log ActivityLog: onboarding_checklist.created + one onboarding_task.created per task.
7. Fire notification onboarding.checklist_started to HR + employee (with portal magic link if applicable).
Error handling + recovery
No matching template: checklist creation skipped, HR notified. Does NOT block employee creation (Hired-stage transition completes either way — onboarding is non-blocking to the hiring decision).
Assignee unresolvable (e.g. dept has no head set): task created with assignee = HR Manager + note "Reassign needed".
Template edited mid-creation: workflow uses template state at workflow START (snapshot). Edits during workflow don't affect in-flight creation.
Partial failure recovery: if workflow fails between steps 4 and 6, checklist exists but tasks partial. Nightly reconciliation finds checklists where tasks_total ≠ count(tasks) and flags for HR review.

  • Draft vs submitted collision: if draft has email = X and an Applicant already exists for (opening, email=X), warn at Step 7: "You've already applied to this role on {date}. Check your status instead." Don't allow duplicate submit.
Form validation strategy
Client-side: per-field validation as user types (email format, required fields, phone format). Step "Next" button disabled while any required field invalid.
Server-side at each auto-save: lightweight — just store what the user typed. No strict validation (let them save partial junk if they want).
Server-side at final submit: full revalidation of ALL fields against schema. If anything fails, reject with per-field error list. Rejecting frustrates users — so client-side must catch 99% of issues before submit.
File validation: mime-type whitelist + size limit + virus scan (ClamAV plugin) for resume/docs. Rejected files shown as inline error on that step.

Draft workflow anti-abuse summary v20 audit-refinement
Three-layer protection:

Layer 1 — CAPTCHA at first save: blocks automated form-farming. Subsequent auto-saves trust the issued draft_token as implicit session — not re-prompted every 30 seconds (would break UX). Token expires after 30 days (draft_token_expires).
Layer 2 — Per-IP + per-token rate limits: 20 saves/hour per IP (broad), 100 saves/hour per draft_token (matches expected auto-save cadence of 30-sec debounce × 60 min = 120, with margin). Rapid resubmission or parallel drafts hit the IP cap.
Layer 3 — Silent failure on load: wrong token returns 404 with no distinguishing error between "expired" and "never existed" — blocks brute-force token enumeration.
Storage cap: max 1 active draft per (IP, opening) for guest path; max 1 per (account, opening) for logged-in path. Second attempt overwrites first's form_data_json (not a new row). Prevents storage-flooding attacks.

Submit application workflow — detailed steps
public.opening.submit_application is the most complex public workflow. Full step-by-step:
1. Validate CAPTCHA, reject if fail.
2. Check honeypot → if populated, create Applicant with honeypot_tripped = yes (for audit) but skip notifications.
3. Validate company_slug → Company (active, tier includes module).
4. Validate opening_slug under company → JobOpening (open, is_public).
5. Rate-limit check per IP (max 5 submissions per hour).
6. Check for duplicate: search Applicants where opening = X AND email = Y. If exists → reject "already applied, check status".
7. Validate all fields server-side (schema, lengths, file types, required consents present).
8. Knockout check: for each screening answer, if matches the question's knockout_answer → flag Applicant for auto-reject.
9. Create Applicant record with all form data + retention_category = "active" + scheduled_deletion_date computed from Company's GDPRSettings.
10. Create child records: ApplicantScreeningAnswer (one per answer), ApplicantEmploymentHistory (one per employment row), ApplicantDocument (one per upload — files already uploaded during draft).
11. If draft_token provided: mark ApplicantDraft.submitted = yes.
12. If session_token provided (logged-in flow): link Applicant.account = ApplicantAccount.
13. Initial stage: Applied. Create ApplicantStageHistory row.
14. If knockout: transition stage → Rejected, reason = "Knockout: {question_text}". Skip HR notification, send polite rejection email to applicant.
15. Else: fire notifications — applicant.submitted to opening.recruiter + opening.hiring_manager (in-app + email); welcome email to applicant.
16. ActivityLog: applicant.created with actor = null, source_ip captured.
17. Increment JobOpening.applications_count.
18. Return { applicant_id, reference_code }.
Rate limiting implementation
Custom RateLimitCounter Data Type: { key, endpoint, window_start, count, created_date }. Key format: ip:{address} or email:{value} or draft:{token}. Every public workflow's step 1 (after basic validation): search for matching (key, endpoint, window_start = today rounded down). If count > threshold, reject with 429-style response. Else increment count or create new row. Scheduled daily cleanup deletes rows older than 7 days. Thresholds per endpoint in a config Data Type (editable by admin).
🔁 Double-submit race guard v20 audit-refinement
Two tabs open, user clicks Submit in both within 100ms. Step 6's duplicate check is a read-then-compare — both tabs can pass simultaneously, then both create Applicants. Protections (defense in depth):

Client-side: Submit button disables on click + shows spinner; page-level custom state submission_in_progress = yes prevents re-firing. Browser back → resubmit prevented by submitted state flag on the form. Mitigates 90% of cases.
Draft-lock (preferred): if a draft_token exists, step 1 marks ApplicantDraft.submission_locked_at = now with "Only when draft.submission_locked_at is empty" guard. Second invocation fails the guard and returns the existing applicant_id from the first submission (idempotent-return pattern). Requires new submission_locked_at field on ApplicantDraft.
Guest (no draft) path: step 6 duplicate check uses "Only when" conditional create: Create Applicant ONLY WHEN Search Applicants (opening = X, email = Y, email_honeypot = no) count = 0. Race still possible in Bubble, so add post-create dedupe sweep (see below).
Post-create dedupe sweep (backup): after step 18, a scheduled 1-minute-delayed "cleanup" backend workflow: for this new Applicant, search for OTHER Applicants with same (opening, email, submitted within ±30 seconds). If found, mark the LATER-created one with is_duplicate_of = earlier one, stage = Withdrawn, reason = "Race-condition duplicate — auto-merged". Does not notify applicant (too noisy for edge case). Preserves audit trail.
Idempotent return: if submit is called with a draft_token that's already been submitted, return the existing {applicant_id, reference_code} from the prior submission — don't create new, don't error. User sees success screen either way.

🌱 GDPRSettings auto-seed workflow (critical — must run before any application can be submitted)
The application submit workflow (step 9) reads Company's GDPRSettings retention periods — if no row exists, the expression returns empty and scheduled_deletion_date calculation fails. Day-1 of a new company signup, there's no GDPRSettings row yet. Fix:

1. Backend workflow "seed_company_gdpr_settings": triggered as step N of the Company signup flow (alongside Division/Section/Department seeding). Creates one GDPRSettings with the defaults above. Idempotent — skip if row already exists for the company.
2. Defensive guard in every retention-reading workflow: if Search GDPRSettings where company = X :first item is empty, call seed_company_gdpr_settings(X) then re-read. Covers legacy companies created before v20 + any edge case where seeding failed.
3. Tier gating: GDPRSettings is seeded regardless of tier. The Onboarding module itself is tier-gated (Executive/Enterprise), but the settings row exists so deferred tier upgrades don't crash on first application.
🗓 Retention sweep — nightly anonymization workflow v20 audit-refinement
Separate from ApplicantErasureRequest nightly sweep. Runs per-company per-timezone at ~02:00 local. Processes retention-expired Applicants:

Query: Search Applicants where company = X AND is_anonymized = no AND scheduled_deletion_date < now AND GDPRSettings.auto_anonymize_enabled = yes. Batch-process 100 rows per pass.
Per row: idempotency guard — abort this row if is_anonymized = yes (another pass already handled it). Then: anonymize Applicant + child records per the anonymization rules (same as erasure); set is_anonymized = yes, anonymized_date = now, anonymization_reason = "Retention expired ({retention_category})"; log ActivityLog applicant.anonymized.
Scheduler idempotency: if the sweep runs twice on the same night (scheduler hiccup), the is_anonymized = yes guard + anonymized_date = now being set transactionally ensures no double-processing.
Manual-mode: when auto_anonymize_enabled = no on GDPRSettings, the sweep only flags overdue records — surfaces them in HR dashboard under "Retention overdue — awaiting review". HR clicks "Anonymize now" per-row or batch.
ApplicantAccount cleanup: after anonymizing per-company, check if the account's Applicants list across ALL companies is fully anonymized — if yes, anonymize the account itself. Runs as a second pass in the same nightly window.

🔀 Fan-out pattern for portal erasure requests
An applicant with accounts at Company A + Company B submits one "delete my data" request via the portal. The data model requires company-scoped rows (each tenant is a separate data controller under GDPR and must process the request independently, with its own SLA clock, legal review, and audit log). Fan-out:

1. Portal workflow portal.gdpr.request_erasure: takes session_token + grounds + reason_text. Resolves ApplicantAccount. Searches Applicants where account = X AND is_anonymized = no, group by company → returns list of distinct companies.
2. Generate single portal_request_id (UUID).
3. For each company: create one ApplicantErasureRequest with company FK set, portal_request_id shared across siblings, status = in_grace. Each row has its OWN grace_expires_date + sla_deadline_date computed from THAT company's GDPRSettings (grace periods may differ).
4. Cancellation: portal.gdpr.cancel_erasure cancels ALL siblings with same portal_request_id — applicant can't selectively un-delete from one tenant.
5. Execution: each sibling executes independently on its own grace schedule. Portal shows per-company status ("Company A: executed 2026-05-02 · Company B: in grace until 2026-05-05").
6. ApplicantAccount itself: anonymized only when ALL siblings reach executed status (or all reach terminal states — cancelled/rejected still leaves account intact for remaining active companies).

Self-service erasure workflow (portal) — fan-out execution
1. Applicant visits /portal/delete-account, sees consequences explanation + grounds selector + free-text reason. Page also shows preview: "This will delete your data from Company A, Company B, Company C" (distinct companies inferred from Applicants).
2. On submit: portal.gdpr.request_erasure:
    a. Validate session_token → ApplicantAccount.
    b. Search Applicants where account = X AND is_anonymized = no. Collect distinct companies list.
    c. Generate one portal_request_id (UUID) to group siblings.
    d. For each company, create an ApplicantErasureRequest row with company FK, shared portal_request_id, grounds, reason_text, status = in_grace, grace_expires_date computed from that company's GDPRSettings.erasure_grace_days (may differ between tenants).
    e. Log activity event applicant.erasure_requested for each company with the shared portal_request_id.
3. Confirmation email — single message listing all affected companies + earliest + latest grace deadline. "Cancel within X days" link uses portal_request_id.
4. User can cancel via portal.gdpr.cancel_erasure(portal_request_id) → sets status = cancelled on ALL sibling rows (atomic — cannot selectively un-delete from one tenant). Each cancellation logs an ActivityLog event.
5. Scheduled nightly workflow (per company, timezone-aware): find requests where company = X AND status = in_grace AND grace_expires_date < now → transition each to approved then executed. Siblings execute INDEPENDENTLY at each company's own schedule — a 14-day grace at Company A and 21-day grace at Company B produce two different execution dates.
6. Execute per company (step must be idempotent — re-running skips if status already = executed):
    a. Anonymize all Applicants where account = X AND company = Y AND is_anonymized = no.
    b. Anonymize child records for those Applicants (documents deleted, notes/messages redacted per anonymization rules above).
    c. Write list of anonymized record IDs to records_anonymized_json for this request row.
    d. Set status = executed, executed_date = now.
7. ApplicantAccount anonymization (deferred): nightly workflow also checks — if all sibling rows for a portal_request_id are in terminal state AND all have status = executed (not cancelled/rejected), then anonymize the ApplicantAccount itself. If any sibling is cancelled/rejected, the account stays active for the remaining tenant relationships.
8. Activity log entries: applicant.anonymized per Applicant (scoped to its company's ActivityLog). applicant_account.anonymized when step 7 fires.
9. Confirmation email to applicant when each company completes + final email when account itself anonymized.
Idempotency guard on execution
The nightly erasure sweep may run twice if scheduling hiccups. Step 6 checks status != executed before anonymizing. Step 7 (account-level) checks ApplicantAccount.is_anonymized = no. Re-running is safe — no double-processing, no extra activity log entries.

Implementation notes
  • Native Bubble rules for internal users: Straightforward Current User's company = Thing's company AND Current User's role contains permission X patterns.
  • Token-only rules for ApplicantAccount access: Bubble's privacy rules CAN'T key off a non-User type. So the Data Type's default rule = "no access" for Everyone AND logged-in users, and all applicant-facing reads go through API workflows that validate session_token server-side before returning data.
  • Public rules: Explicit "Everyone" access rule with the exact field-level whitelist from the Public-Access Privacy Pattern (Phase 1 core principles).
  • Test coverage: manual test checklist — for each DT, verify anonymous access, logged-in-applicant access (correct + cross-account), internal-user access (with + without permission). Automated testing of privacy rules is not well-supported in Bubble — rely on thorough manual testing of release candidates in incognito windows.
